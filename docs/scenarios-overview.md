# SKILL.md 场景梳理与情况重构

> 日期: 2026-05-12
> 状态: 已实施（SKILL.md 已重构为三层结构）

## 背景

Round 10 E2E 测试发现 G4（跨 Batch 混合 spawn）根因是 SKILL.md 的编排指引不够清晰。在修复 G4 过程中发现情况 A 场景覆盖不完整，需要重新梳理所有场景并重构 SKILL.md 的编排结构。

## 两个维度

**学校数量**：单校 / 多校（含"一个国家"筛选版） / 全部

**获取方式**：三阶段流程 / 增量更新 / 单页面提取 / 发现新学校

### 排列组合矩阵

|  | 单校 | 多校（指定） | 多校（条件筛选） | 全部 |
|--|------|-------------|-----------------|------|
| **三阶段流程（爬取）** | ✅ 爬取 XX | ✅ 爬取 XX、YY | ❌ 不合理 | ❌ 不合理 |
| **三阶段流程（重爬）** | ✅ 重新爬取 XX | ✅ 重新爬取 XX、YY | ✅ 重爬最旧的 3 所 | ✅ 全量重爬 |
| **增量更新** | ✅ 更新 XX 数据 | ✅ 更新 XX、YY 数据 | ✅ 更新 fill-rate 最低的 5 所 | ✅ 更新全部 |
| **单页面提取** | ✅ 提取这个 URL | ❌ | ❌ | ❌ |
| **发现新学校** | ✅ 搜索 XX 学校 | ✅ 搜索德国设计院校 | ❌ | ❌ |

> 注：多校（条件筛选）和全部可附加国家筛选（如"更新所有英国学校"= 多校筛选的一种）。

## 更新范围标注

每种情况影响的数据层级：

| 数据层级             | 三阶段流程            | 增量更新                                           | 单页面提取 |
| -------------------- | --------------------- | -------------------------------------------------- | ---------- |
| site_map.md          | ✅ 覆盖重建           | ❌ 不动                                            | ❌ 不动    |
| 新专业发现           | ✅ site-explorer 重扫 | ⚠️ smart-extractor 可发现新子页面，补充到 site_map | ❌         |
| program \_index 文件 | ✅ 全部重新提取+翻译  | ✅ 增量提取已有 URL 的数据                         | ✅ 单页面  |
| university \_index   | ✅ Phase 3 聚合       | ✅ 更新 profile/fill-rate/tags                     | ❌ 不动    |
| crawl_state          | ✅ 重置               | ✅ 更新                                            | ✅ 更新    |

**关键区别**：

- **三阶段流程**：site_map 覆盖重建 → 可能发现新增/删除的专业 → 全部 program 重新提取
- **增量更新**：保留现有 site_map → 更新 program 数据 → 完成后刷新 university \_index（profile、fill-rate、tags 聚合）
- **单页面提取**：只处理一个 URL，不影响其他数据

## 条件筛选说明

"条件筛选"场景不需要写专门的脚本或代码，LLM 读取 `collection_status.yaml` 后自行排序筛选即可。可行的筛选条件举例：

- `last_synced` 最早的 N 所（最久未更新）
- `field_fill_rate` 最低的 N 所（数据质量最差）
- `programs_total` 最多的 N 所（专业最多的）
- `last_explored` 最早的 N 所（最久未重爬）
- 某个国家 + 上述任意条件的组合

## SKILL.md 新结构：操作过程 + 场景引用

不按 A/B/C/D/E 分类，而是分三层：

### 第一层：判断路由（伪代码）

收到用户请求后，按优先级匹配场景，确定操作过程和学校范围。

### 第二层：场景表（12 个场景的快速参考）

每个场景一行，标明：触发条件 → 前置步骤 → 操作过程 → 编排方式 → 后置步骤

```
| 用户意图                   | 前置           | 操作过程   | 编排         | 后置          |
| "爬取 XX"                  | auto detect    | 三阶段     | N/A          | commit        |
| "爬取 XX、YY"              | auto each      | 三阶段     | 批次调度     | commit each   |
| "重新爬取 XX"              | reset          | 三阶段     | N/A          | commit        |
| "重新爬取 XX、YY"          | reset each     | 三阶段     | 批次调度     | commit each   |
| "重爬 fill-rate 最低的 5 所" | 筛选+reset   | 三阶段     | 批次调度     | commit each   |
| "全量更新"                  | reset --all    | 三阶段     | 遍历+批次    | commit + push |
| "更新 XX 数据"              | 无             | 增量更新   | N/A          | commit+push   |
| "更新 XX、YY 数据"          | 无             | 增量更新   | 可并行       | commit+push   |
| "更新最旧的 3 所"           | 筛选           | 增量更新   | 可并行       | commit+push   |
| "更新全部"                  | 无             | 增量更新   | 遍历         | commit + push |
| "提取这个 URL"              | 无             | 单页面提取 | N/A          | commit+push   |
| "搜索设计院校"              | 无             | 发现新校   | N/A          | → 确认后爬取  |
```

### 第三层：操作过程详细定义（可复用）

**操作过程 1：三阶段流程**
- 前置变体：init（未收录）/ reset（已收录/重爬）/ auto detect（自动判断）
- Phase 1: site-explorer → site_map.md
- Phase 2: smart-extractor × N → program _index_EN/ZH/DE
- Phase 3: data-organizer → aggregate + profile + fill-rate
- 多校时遵循"批次调度规范"

**操作过程 2：增量更新**
- 读取 site_map.md → smart-extractor 增量提取已有 URL 数据
- 完成后：aggregate_tags + validate --fix + validate --fill-rate → 更新 university profile
- 多校时可并行

**操作过程 3：单页面提取**
- smart-extractor 单页面模式

**操作过程 4：发现新学校**
- university-scout → 确认后路由到对应场景

**共享规范：批次调度规范**
- 严格批次隔离
- HEARTBEAT.md 操作规则
- 恢复机制

**共享规范：条件筛选**
- 读 collection_status.yaml → 按用户指定字段排序 → 取前 N 所
- 不需要专门脚本，LLM 自行处理

### 与当前结构的对比

| 维度 | 当前（A/B/C/D/E） | 新结构（操作过程+场景引用） |
|------|-------------------|--------------------------|

## 备注：触发方式

上述场景的触发方式有两种：

1. **用户消息**：用户在飞书频道或 CLI 中发送请求，LLM 按判断路由匹配场景并执行
2. **定时触发**（未来支持）：通过 cron 或 heartbeat 定时检查 `collection_status.yaml` 中的到期字段，对到期院校自动执行对应操作（`next_sync` 到期 → 增量更新，`next_explore` 到期 → 三阶段重爬）。当前框架暂不支持定时触发，此功能待后续实现
